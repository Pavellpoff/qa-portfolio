# Test Cases (Full)

## 1. Позитивный сценарий

### 1.1 Регистрация с валидными данными
**Предусловие:**
- email: новый (например, test1@mail.ru)
- name: валидный (например, Ivan)
- password: валидный (например, Test123)

**Шаги:**
1. Отправить запрос на регистрацию
2. Проверить ответ API
3. Проверить пользователя в GUI
4. Попробовать авторизоваться

**Ожидаемый результат:**
- API возвращает HTTP 200
- в ответе возвращается объект пользователя
- email и name в ответе совпадают с отправленными
- пароль сохраняется в виде хэша (не равен исходному значению)
- пользователь отображается в GUI
- авторизация с этими данными успешна

**Фактический результат:**
- HTTP 200
- пользователь создан
- авторизация успешна

## 2. Обязательные поля

### 2.1 Пустой email
**Шаги:**
1. Отправить запрос с пустым email

**Ожидаемый результат:**
- API возвращает HTTP 400

- сообщение об ошибке содержит орфографическую ошибку

## 5. Валидация name

### 5.1 Name с пробелами по краям
**Пример:** " Сергей "

**Предусловие:**
- email: новый
- name: " Сергей "
- password: валидный

**Шаги:**
1. Отправить запрос на регистрацию
2. Проверить значение name в ответе API
3. Проверить отображение name в GUI

**Ожидаемый результат:**
- пробелы в начале и конце строки обрезаются (trim)
- пользователь создаётся
- значение name сохраняется в нормализованном виде (без лишних пробелов)
- данные в API и GUI совпадают

**Фактический результат:**
- пользователь создаётся
- в API: name = "Сергей" (пробелы обрезаны)
- в GUI: name отображается без пробелов

**Вывод:**
- система корректно обрабатывает пробелы по краям (trim)
- нормализация данных выполняется корректно

---

### 5.2 Name, состоящий только из пробелов
**Пример:** "   "

**Предусловие:**
- email: новый
- name: "   "
- password: валидный

**Шаги:**
1. Отправить запрос на регистрацию с name из пробелов
2. Проверить ответ API

**Ожидаемый результат:**
- значение считается пустым после trim
- API возвращает HTTP 400
- сообщение об ошибке указывает на обязательность поля name
- пользователь не создаётся

**Фактический результат:**
- HTTP 200 ❌
- сообщение: "поле фио является обязательным"
- пользователь не создан

**Вывод:**
- система корректно определяет значение как пустое (после trim)
- но:
  - возвращает некорректный HTTP статус
  - использует некорректную терминологию ("ФИО" вместо "name")

---

### 5.3 Name с несколькими пробелами внутри
**Пример:** "Александр   Петров"

**Предусловие:**
- email: новый
- name: "Александр   Петров"
- password: валидный

**Шаги:**
1. Зарегистрировать пользователя
2. Проверить значение name в API
3. Проверить отображение name в GUI

**Ожидаемый результат:**
- пробелы по краям обрезаются
- внутренние пробелы:
  - либо сохраняются как есть
  - либо нормализуются (например, до одного пробела)
- данные в API и GUI совпадают

**Фактический результат:**
- пользователь создаётся
- API: name = "Александр   Петров" (без крайних пробелов)
- GUI: name = " Александр   Петров " ❌

**Вывод:**
- система частично нормализует значение (trim только по краям)
- обнаружена неконсистентность:
  - API и GUI отображают разные значения одного и того же поля

---

### 5.4 Спецсимволы в name
**Пример:** "t&*(+/"

**Предусловие:**
- email: новый
- name: "t&*(+/"
- password: валидный

**Шаги:**
1. Отправить запрос на регистрацию с невалидным name
2. Проверить ответ API

**Ожидаемый результат:**
- API возвращает HTTP 400
- сообщение об ошибке указывает на некорректный формат name
- пользователь не создаётся

**Фактический результат:**
- HTTP 500 ❌
- сообщение: "Bad Request"
- пользователь не создан

**Вывод:**
- сервер некорректно обрабатывает невалидные входные данные
- возвращается HTTP 500 вместо ошибки валидации (4xx)
- сообщение "Bad Request" не соответствует статусу 500
- отсутствует корректная обработка ошибок на стороне сервера
