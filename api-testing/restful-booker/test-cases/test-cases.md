# API Test Cases - Restful Booker

Данный документ содержит тест-кейсы для проверки API сервиса Restful Booker.

**Покрытие:**

* Auth (CreateToken)
* Booking (GetBookingIds)
* Booking (GetBooking)
* Booking (CreateBooking)
* Booking (UpdateBooking)


**Инструменты:**

* Postman

---

## 1. CreateToken с валидными данными

**Метод:** POST /auth

**Тип:** Позитивный

**Предусловие:**
API доступно

**Шаги:**

1. Отправить POST запрос на `/auth`
2. В теле запроса передать:

```json
{
  "username": "admin",
  "password": "password123"
}
```

**Ожидаемый результат:**

* HTTP 200 OK
* В ответе присутствует поле `token`
* Тип значения `token` — string
* Значение `token` не пустое

## 7. Получение списка booking ID по фильтру checkin/checkout

**Метод:** GET /booking

**Тип:** Позитивный

**Предусловие:**
API доступно
В системе существуют бронирования с заданными датами


### Шаги

1. Отправить GET запрос на
   `/booking?checkin=2016-07-05&checkout=2022-01-09`


### Ожидаемый результат

* HTTP 200 OK
* Тело ответа представляет собой массив
* Массив не пустой (если данные существуют)
* Каждый элемент массива — объект
* Каждый объект содержит поле `bookingid`
* Тип значения `bookingid` — number


### Пример ответа

```json id="e7h3vd"
[
  { "bookingid": 7 },
  { "bookingid": 11 }
]
```


### Примечание

Для полной проверки корректности фильтрации необходимо:

* выполнить GET `/booking/{id}` для каждого полученного `bookingid`
* убедиться, что даты `checkin` и `checkout` соответствуют заданным параметрам

## 8. Получение списка booking ID с некорректными значениями checkin/checkout

**Метод:** GET /booking

**Тип:** Негативный

**Предусловие:**
API доступно


### Шаги

1. Отправить GET запрос на `/booking` с некорректными параметрами:

   * `checkin="", checkout = 2022-01-09`
   * `checkin = 05.07.2019, checkout=""`


### Ожидаемый результат

* API корректно обрабатывает невалидные входные данные
* Возможные варианты поведения:
  * HTTP 400 Bad Request с описанием ошибки
  * или HTTP 200 OK с пустым массивом
* API не должен возвращать ошибку (5xx)


### Примечание

Некорректные форматы дат и пустые значения должны валидироваться на стороне API.

## 9. Получение списка booking ID с невалидными числовыми значениями checkin/checkout

**Метод:** GET /booking

**Тип:** Негативный

**Предусловие:**
API доступно


### Шаги

1. Отправить GET запрос на `/booking` с параметрами:

   * `checkin = 2356, checkout = 2023-01-09`
   * `checkin = 05.07.2019, checkout = 0`
   * `checkin = 0, checkout = 2023-01-09`


### Ожидаемый результат

* API корректно обрабатывает невалидные значения параметров
* Возможные корректные варианты поведения:

  * HTTP 400 Bad Request
  * или HTTP 200 OK с пустым массивом
* Результат должен быть консистентным для всех невалидных значений


### Примечание

Требуется уточнение требований к обработке некорректных числовых значений в параметрах даты (`checkin`, `checkout`).


### Наблюдение

Поведение API может отличаться в зависимости от переданных значений (например, `checkin = 0` может возвращать данные).
Это может указывать на некорректную интерпретацию параметров.

**Тип:** Позитивный

**Предусловие:**
API доступно


* HTTP 200 OK
* В ответе присутствует `bookingid` (number)
* В ответе присутствует объект `booking`

**GET:**

* HTTP 200 OK
* Данные соответствуют отправленным
* Структура ответа корректная

## 15. Обновление существующего бронирования с валидной авторизацией

**Метод:** PUT /booking/{id}

**Тип:** Позитивный

**Предусловие:**

* Создано бронирование через POST `/booking`
* Получен `bookingid`
* Проверено через GET `/booking/{id}`, что данные сохранены корректно
* Получен токен через POST `/auth`


### Шаги

1. Отправить PUT запрос на `/booking/{id}`
2. Добавить header:

```
Cookie: token={valid_token}
```

3. В теле запроса передать обновлённые данные:

```json id="k9d2qm"
{
  "firstname": "James",
  "lastname": "Bond",
  "totalprice": 189,
  "depositpaid": false,
  "bookingdates": {
    "checkin": "2020-03-01",
    "checkout": "2022-04-01"
  },
  "additionalneeds": "Dinner"
}
```

4. Отправить GET запрос на `/booking/{id}`


### Ожидаемый результат

* HTTP 200 OK
* Все поля бронирования обновлены
* `bookingid` не изменяется
* Структура ответа корректная

**GET:**

* Возвращаются обновлённые значения
* Данные соответствуют отправленным

**Метод:** PUT /booking/{id}

**Тип:** Негативный

**Предусловие:**

* Существует бронирование (`bookingid`)
* Проверено через GET `/booking/{id}`


### Шаги

1. Отправить PUT запрос на `/booking/{id}` без корректной авторизации:

  **или**

### Примечание

Доступ к обновлению бронирования должен быть ограничен и требовать валидную авторизацию.


### Пример ответа (фактическое поведение)

```json id="v7c3xp"
HTTP 403 Forbidden
```
