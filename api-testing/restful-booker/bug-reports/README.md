## BUG 1. API возвращает 200 OK при невалидной авторизации

**Метод:** POST /auth

**Серьезность:** Средняя


### Описание

API возвращает успешный HTTP-статус (200 OK) при передаче невалидных данных авторизации.
При этом в теле ответа содержится сообщение об ошибке.

Это противоречит REST-практикам, где ошибки авторизации должны сопровождаться статусами 4xx.


### Предусловие

API доступно


### Шаги воспроизведения

1. Отправить POST запрос на `/auth`
2. В теле запроса передать невалидные данные:

```json
{
  "username": "admin",
  "password": "wrong_password"
}
```


### Ожидаемый результат

* HTTP 401 Unauthorized (или 400 Bad Request для некорректных данных)
* В теле ответа присутствует сообщение об ошибке


### Фактический результат

* HTTP 200 OK
* Тело ответа:

```json
{
  "reason": "Bad credentials"
}
```


### Окружение

* Base URL: https://restful-booker.herokuapp.com/
* Инструмент: Postman


### Влияние

Клиентские приложения не могут корректно определять ошибки авторизации по HTTP-статусу.
Это может привести к неправильной обработке ответов и логическим ошибкам на стороне клиента.

## BUG 2: API возвращает 500 Internal Server Error при невалидных параметрах checkin/checkout

**Метод:** GET /booking

**Серьезность:** Высокая


### Описание

При передаче некорректных значений параметров `checkin` и `checkout` API возвращает ошибку сервера (500 Internal Server Error) вместо корректной обработки входных данных.


### Предусловие

API доступно


### Шаги воспроизведения

1. Отправить GET запрос на `/booking` с параметрами:

   * `checkin="", checkout = 2022-01-09`
   * `checkin = 05.07.2019, checkout=""`


### Ожидаемый результат

* HTTP 400 Bad Request с описанием ошибки
* или HTTP 200 OK с пустым результатом


### Фактический результат

* HTTP 500 Internal Server Error


### Окружение

* Base URL: https://restful-booker.herokuapp.com/
* Инструмент: Postman


### Влияние

Некорректная обработка входных данных приводит к серверной ошибке.
Это может вызвать нестабильность API и потенциально привести к отказу сервиса при массовых запросах.

## BUG 3: Некорректная обработка невалидных значений дат (checkin/checkout) при создании бронирования

**Метод:** POST /booking

**Серьезность :** Высокая


### Описание

При передаче невалидных значений в поля `checkin` и `checkout` API не отклоняет запрос, а сохраняет некорректные данные в формате `0NaN-aN-aN`.


### Предусловие

API доступно


### Шаги воспроизведения

1. Отправить POST запрос на `/booking` с телом:

```json id="n3u8cj"
{
  "firstname": "Jim",
  "lastname": "Brown",
  "totalprice": 111,
  "depositpaid": true,
  "bookingdates": {
    "checkin": "",
    "checkout": "2026-02-01"
  },
  "additionalneeds": "Breakfast"
}
```

2. Повторить с вариантами:

   * `checkin="ab"`
   * `checkin=" "`
   * аналогично для `checkout`


### Ожидаемый результат

* HTTP 400 Bad Request
* Некорректные значения не сохраняются


### Фактический результат

* HTTP 200 OK
* В ответе:

```json id="4q9k2a"
"checkin": "0NaN-aN-aN"
```


### Окружение

* Base URL: https://restful-booker.herokuapp.com/
* Инструмент: Postman


### Влияние

* Нарушение целостности данных
* Некорректная работа фильтрации по датам
* Возможные ошибки в бизнес-логике

## BUG 4: API возвращает 403 вместо 400 при отсутствии обязательных полей

**Метод:** PUT /booking/{id}

**Серьезность:** Средняя


### Описание

При обновлении бронирования без обязательных полей (`totalprice`, `lastname`) API возвращает статус 403 Forbidden вместо ожидаемого 400 Bad Request.


### Предусловие

* Существует бронирование
* Передан валидный токен


### Шаги воспроизведения

1. Отправить PUT `/booking/{id}`
2. Передать валидный `Cookie: token`
3. Удалить одно из обязательных полей:

   * `totalprice`
   * `lastname`


### Ожидаемый результат

* HTTP 400 Bad Request
* Сообщение о некорректных входных данных


### Фактический результат

* HTTP 403 Forbidden


### Влияние

Клиентские приложения некорректно интерпретируют ошибку как проблему авторизации, а не валидации данных.

## BUG 5: API возвращает 403 вместо 404 при обновлении несуществующего ресурса

**Метод:** PUT /booking/{id}
**Серьезность:** Средний


### Описание

При попытке обновить несуществующее бронирование API возвращает 403 Forbidden вместо ожидаемого 404 Not Found.


### Предусловие

* Бронирование с указанным `bookingid` не существует
* Передан валидный токен


### Шаги воспроизведения

1. Отправить PUT `/booking/999999`
2. Передать валидный `Cookie: token`


### Ожидаемый результат

* HTTP 404 Not Found


### Фактический результат

* HTTP 403 Forbidden


### Влияние

Клиент не может корректно различать:

* отсутствие ресурса
* и проблемы с доступом

Это усложняет обработку ошибок и логику клиента.
