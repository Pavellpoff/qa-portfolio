## 1. Создание пользователя с несуществующими tasks
**Предусловие:**
- email: testh127@mail.ru
- name: Павел
- tasks: 5
- companies: 3, 4
Используются произвольные ID задач и компаний (валидность не подтверждена). 

**Тип теста:** Негативный

**Причина:** ID задач и компаний неизвестны - используем предположительные id. 



**Ожидаемый результат:**
**API:**
- API возвращает ошибку (HTTP 4xx или явный признак ошибки в теле ответа)
пользователь не создается
**GUI:**
пользователь не создается

**Фактический результат:**
**API:**
- HTTP статус = 200 (ожидался 400)
- вернулось тело ответа с сообщением об ошибке: задачи не найдены
- валидация прерывается на проверке tasks, поведение для companies не проверено
  
**GUI:**
- пользователь не создан.
  
**Вывод.** Сервис корректно обрабатывает невалидные ID задач. 

**Обнаружены ошибки:**
- API возвращает HTTP 200 при бизнес-ошибке. Ожидалось: 400. 
- Ответ API не содержит признака ошибки (status/error code) 
- Нужен отдельный тест-кейс с проверкой на валидацию companies 
В документации не известны id задач и компаний для корректного их прикрепления к пользователям. 


## 2. Создание пользователя по примеру из документации 
**Предусловие:**
Используются ID:
- tasks = 12 (валидный, взят из примера документации)
- companies = 36 (валидный, взят из примера документации)

Данные пользователя:
- email: testh166@mail.ru
- name: Антон

**Тип теста:** Смешанный (позитивный + негативный)

**Описание:** Проверка поведения системы при передаче частично валидных данных: корректный ID задачи и потенциально невалидный ID компании.

**Ожидаемый результат:**

**API:**

Если все данные валидны:
- HTTP 200
- пользователь создан
- tasks и companies привязаны

Если часть данных невалидна:
или: HTTP 4xx, пользователь не создается
или: пользователь создается, но API возвращает явный признак ошибки (status/error)
**GUI:**
- данные соответствуют результату API
- при ошибке отображается корректное сообщение

**Фактический результат:**

**API:**
- HTTP статус: 200
- пользователь создан
- tasks применился (id = 12)
- companies не применился
- поле companies пустое
- в ответе отсутствует явный признак ошибки
- значения birthday и date_start = 0

**GUI:**
- пользователь создан
- tasks отображается “Расписание”
- companies отсутствует
- отображается сообщение: “Компании не найдены”

**Вывод:**
Пример в документации не соответствует фактическому поведению системы: указанный ID компании не является валидным, это приводит к частично некорректному выполнению операции. 
Сервис некорректно обрабатывает частично невалидные данные:
операция выполняется частично (пользователь создается без привязки компаний)
API не сигнализирует об ошибке
наблюдается рассинхронизация между API и GUI
нарушается целостность операции

## 5. Создание пользователя с пустыми значениями для необязательных полей
**Предусловие:**

Используются ID:
 - tasks = 12 (валидный, взят из примера документации)
 - companies = 37 (валидный, взят из примера документации)
**Данные пользователя:**
 - email: test007@mail.ru
 - name: Николай
 - значение всех необязательных полей: “”.

**Описание:** Негативный тест на проверку валидации необязательных полей. 

**Ожидаемый результат:**

**API:**

- API возвращает ошибку валидации
- HTTP статус: 4xx или 200 с явным признаком ошибки в теле ответа
- пользователь не создается

**GUI:**

- пользователь не создается

**Фактический результат:**

**API:**
- HTTP статус: 200
- Вернулось тело ответа с сообщением об ошибке: Значение  некорректное,должна быть дата в формате ДД.ММ.ГГГГ
- В сообщении об ошибке не указано конкретное поле (birthday или date_start). Присутствуют орфографические ошибки 
явный признак ошибки отсутствует  (status/error)

**GUI:**
пользователь не создан

**Вывод:**
- Система корректно отклоняет невалидные значения в необязательных полях типа “дата”, но реализация обработки ошибок некорректна:
сообщение об ошибке не содержит указания на конкретное поле
- присутствуют орфографические ошибки
- отсутствует структурированный признак ошибки
