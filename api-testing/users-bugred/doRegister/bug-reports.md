# Bug reports для метода doRegister

## Bug 1. API возвращает HTTP 200 при ошибках валидации входных данных##
Описание:
Сервис регистрации пользователей возвращает HTTP статус 200 OK для всех запросов, включая случаи с невалидными входными данными.
При этом информация об ошибке передается только в теле ответа, это может приводить к некорректной обработке ответа на стороне клиента.

Окружение:
- API: users.bugred.ru
- Метод: doRegister
- Инструмент: SoapUI 

Шаги:
1. Отправить запрос с пустым email
2. Отправить запрос с невалидным email (например: "test")
3. Отправить запрос с уже существующим email

Ожидаемый результат:
- При невалидных данных API возвращает HTTP статус 4xx (например, 400)
- В теле ответа содержится сообщение об ошибке
- Пользователь не создается

Фактический результат:
- API всегда возвращает HTTP статус 200 OK
- В теле ответа содержится ошибка (например: "Некорректный email")
- Пользователь не создается

Дополнительно:
Проблема воспроизводится во всех негативных сценариях:
- пустые обязательные поля
- невалидный email
- дубликат email
- некорректные значения name

Влияние:
- Клиентские приложения не могут корректно обрабатывать ошибки
- Возможны ошибки в логике обработки на фронтенде

Серьезность: Средняя / Высокая

## Bug 2. Регистрация пользователя с пустым паролем приводит к созданию невалидного аккаунта
Описание:
Сервис допускает регистрацию пользователя с пустым паролем. Пользователь успешно создается, но авторизация с такими данными невозможна.

Это приводит к созданию невалидных аккаунтов и неконсистентному поведению системы.

Окружение:
- API: users.bugred.ru
- Метод: doRegister
- Инструмент: SoapUI

Шаги воспроизведения:
1. Отправить запрос на регистрацию с пустым паролем
2. Убедиться, что пользователь создан
3. Попробовать авторизоваться с тем же email и пустым паролем

Ожидаемый результат:
- При невалидных данных API возвращает HTTP статус 4xx (например, 400)
- В теле ответа содержится сообщение об ошибке
- Пользователь не создается

Фактический результат:
- API возвращает HTTP статус 200 OK ❌
- пользователь успешно создается
- пароль сохраняется в виде хэша
- авторизация с пустым паролем невозможна

Влияние:
- создание неиспользуемых аккаунтов
- неконсистентность логики системы (регистрация успешна, вход невозможен)
- потенциальное засорение БД
Серьезность: высокая

## Bug 3. Неконсистентная обработка пробелов во входных данных (password, email, name) и расхождение между API и GUI
Название:
Неконсистентная обработка пробелов во входных данных (password, email, name) и расхождение между API и GUI

Описание:
Система по-разному обрабатывает пробелы в полях (password, email, name), и есть расхождение между данными, возвращаемыми API, и отображаемыми в GUI.
Это приводит к неконсистентному поведению системы и созданию невалидных данных.

Окружение:
- API: users.bugred.ru
- Метод: doRegister
- Инструмент: SoapUI

Шаги воспроизведения:
1. Зарегистрировать пользователя с password = "   ". Попробовать авторизоваться = вход невозможен
2. Зарегистрировать пользователя с password = " r23 ". Попробовать авторизоваться → вход успешен (пробелы обрезаны).

3. Зарегистрировать пользователя с email = " test@mail.ru ". Получаем ошибку (пробелы не обрезаются)
4. Зарегистрировать пользователя с name = " Сергей ". В API name = "Сергей", в GUI остаются пробелы. 

Ожидаемый результат:
- пробелы во входных данных обрабатываются единообразно: или обрезаются или значение считается невалидным.
- поведение одинаковое для всех полей (email, name, password)
- данные в API и GUI совпадают

Фактический результат:
password:
  - "   " = пользователь создается, но авторизация невозможна
  - " r23 " = пробелы обрезаются, авторизация успешна
email:
  - пробелы не обрабатываются → ошибка
name:
  - в API пробелы обрезаются
  - в GUI пробелы сохраняются

Вывод:
- отсутствует единообразная обработка пробелов
- обнаружена неконсистентность между API и GUI
- система допускает создание невалидных аккаунтов

Влияние:
- создание неиспользуемых аккаунтов
- рассинхронизация данных между API и интерфейсом
- потенциальные проблемы при поиске и обработке данных

Серьезность: высокая
