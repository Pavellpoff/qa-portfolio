# BAG 1. API позволяет обновлять профиль пользователя невалидными значениями обязательных полей

## Описание

Эндпоинт обновления профиля пользователя принимает и сохраняет невалидные значения обязательных полей без выполнения серверной валидации.

При этом пользовательский интерфейс содержит проверки корректности ввода и не позволяет сохранить аналогичные данные через форму редактирования профиля.

Между клиентской и серверной валидацией наблюдается несогласованность.

---

## Предусловия

* Пользователь **Reyes.Osinski** существует.
* Пользователь успешно авторизован через API.

---

## Шаги воспроизведения

1. Выполнить `POST /login`.
2. Выполнить `PATCH /users/{id}`.
3. Передать, например, следующие данные:

```json
{
  "id": "M1ty1gR8B3",
  "firstName": "",
  "lastName": "",
  "email": "invalid-email",
  "phoneNumber": "",
  "defaultPrivacyLevel": "private"
}
```

4. Выполнить `GET /users/{id}`.
5. Авторизоваться через пользовательский интерфейс.
6. Открыть страницу **My Account**.

---

## Фактический результат

* API возвращает **204 No Content**.
* Невалидные данные успешно сохраняются.
* После входа через UI профиль содержит невалидные значения.
* При попытке повторно сохранить профиль через UI отображаются сообщения о некорректности данных.

---

## Ожидаемый результат

* API должен выполнять серверную валидацию входных данных.
* При передаче некорректных значений сервер должен возвращать ошибку валидации (например, **400 Bad Request** или **422 Unprocessable Entity**).
* Некорректные данные не должны сохраняться.

---

## Дополнительная информация

Проблема подтверждена для следующих сценариев:

* пустой `firstName`;
* пустой `lastName`;
* пустой `email`;
* некорректный `email`;
* пустой `phoneNumber`;
* некорректный `phoneNumber`.

---

## Примечание

UI и API используют разные правила проверки данных.

Через пользовательский интерфейс сохранить такие значения невозможно, тогда как API принимает и сохраняет их без ошибок. Это может привести к появлению неконсистентных данных в системе.
