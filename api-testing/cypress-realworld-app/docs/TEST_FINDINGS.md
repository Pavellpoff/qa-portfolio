# Наблюдения во время тестирования

В данном документе собраны интересные особенности поведения приложения, обнаруженные в процессе исследовательского тестирования. Не каждое наблюдение является подтвержденным дефектом - некоторые представляют собой особенности реализации или потенциальные проблемы пользовательского интерфейса.

---

## FINDING 1. Кнопка **Sign In** становится активной после обновления страницы

### Модуль

Авторизация

### Сценарий

Во время тестирования страницы авторизации было обнаружено нестабильное поведение кнопки **Sign In**.

После нескольких обновлений страницы (**F5**) кнопка может становиться активной, несмотря на то, что поля **Username** и **Password** остаются пустыми.

### Шаги воспроизведения

1. Открыть страницу `/signin`.
2. Убедиться, что поля **Username** и **Password** пустые.
3. Несколько раз обновить страницу (F5).
4. Проверить состояние кнопки **Sign In**.

### Ожидаемый результат

Кнопка **Sign In** должна оставаться неактивной, пока обязательные поля формы не заполнены.

### Фактический результат

После нескольких обновлений страницы кнопка может перейти в активное состояние при пустых полях ввода.

### Дополнительная проверка

После нажатия на активную кнопку было установлено:

- запрос `POST /login` не отправляется;
- отображаются сообщения валидации:
  - `Username is required`;
  - `Password is required`.

### Анализ

Функциональность авторизации работает корректно — приложение не позволяет отправить пустую форму.

Наблюдаемое поведение относится только к визуальному состоянию кнопки и указывает на возможную рассинхронизацию между отображением интерфейса и логикой валидации формы.

### Заключение

Наблюдение не влияет на работоспособность механизма авторизации, но может ухудшать пользовательский опыт и требует дополнительной проверки разработчиками.

## FINDING 2. Отсутствует валидация полей имени и фамилии

### Модуль

Личный кабинет

### Сценарий

Во время тестирования формы редактирования профиля было установлено, что поля **First Name** и **Last Name** принимают значения, состоящие из цифр, специальных символов и длинных строк.

### Проверка

Были введены следующие значения:

- First Name: `44444`
- Last Name: `abcdefgabcdefg abcdefgabcdefg 44/*`

После сохранения данные успешно отображаются в профиле пользователя.

### Анализ

Приложение не выполняет проверку допустимого формата имени и фамилии.

### Заключение

Если бизнес-требования предполагают использование только буквенных символов, такое поведение может рассматриваться как потенциальный дефект. Если ограничения отсутствуют, поведение допустимо.

## FINDING-3. Недостаточно информативный ответ API при пустых учетных данных

### Модуль

Авторизация (API)

### Сценарий

При отправке запроса авторизации с различными некорректными значениями обязательных полей сервер во всех случаях возвращает одинаковый ответ:

Проверенные сценарии:

- `username = ""`, `password = ""`;
- `username = ""`, корректный `password`;
- отсутствует поле `username`;
- `username = null`, `password = null`.

Во всех случаях сервер возвращает:

- HTTP 400;
- `Bad Request`.

### Анализ

Ответ не содержит информации о причине ошибки.

Из него невозможно определить:

- отсутствует имя пользователя;
- отсутствует пароль;
- отсутствуют оба поля;
- данные имеют неверный формат.

### Заключение

Возвращаемый ответ достаточен для обработки ошибки на уровне HTTP, но недостаточно информативен для клиента и затрудняет диагностику причины ошибки.

## FINDING 4. Банковский счет может быть создан со значениями, состоящими только из пробелов

### Модуль
Банковские счета

### Сценарий
Во время тестирования формы создания банковского счета было установлено, что поля **Routing Number** и **Account Number** принимают значения, состоящие только из пробелов.

После отправки формы банковский счет успешно создается и отображается в списке.

### Проверка

Были введены следующие значения:

- Bank Name: `Bank Test`
- Routing Number: `"         "`
- Account Number: `"         "`

После сохранения новый банковский счет успешно отображается в списке.

### Анализ

Пользовательский интерфейс не отклоняет значения, состоящие только из пробелов, и позволяет создать банковский счет.

Такое поведение может свидетельствовать об отсутствии проверки содержимого полей **Routing Number** и **Account Number** или об автоматическом преобразовании введенных данных перед сохранением.

### Заключение

Если бизнес-требования предполагают, что поля **Routing Number** и **Account Number** должны содержать только цифровые значения, такое поведение может рассматриваться как потенциальный дефект.

Если подобные значения допускаются требованиями, поведение является допустимым.

## FINDING 5. Поле Description принимает значение, состоящее только из пробельных символов

### Модуль

Создание запроса на перевод денежных средств

### Сценарий

Во время тестирования формы создания запроса на перевод было установлено, что поле **Description** принимает значение, состоящее только из пробельных символов.

Несмотря на то, что поле является обязательным и не позволяет отправить полностью пустое значение, строка, содержащая только пробелы, успешно проходит клиентскую валидацию и позволяет создать запрос.

### Проверка

Были выполнены следующие действия:

- введена корректная сумма перевода;
- в поле **Description** введен один пробельный символ;
- форма успешно отправлена;
- запрос на перевод успешно создан.

### Анализ

Текущее поведение может свидетельствовать об отсутствии проверки значения поля после удаления пробельных символов (`trim`).

Но без бизнес-требований невозможно однозначно определить, должно ли такое значение считаться пустым.

### Вопрос к требованиям

Должно ли значение поля **Description**, состоящее только из пробельных символов, рассматриваться как пустое и отклоняться валидацией?

### Заключение

Наблюдение требует уточнения бизнес-требований. В случае если значения, состоящие только из пробельных символов, считаются пустыми, такое поведение стоит рассматривать как дефект валидации.

## FINDING 6. Особенность работы фильтра по диапазону суммы

### Модуль

Лента транзакций

### Сценарий

Во время тестирования фильтра по диапазону суммы было установлено, что фильтрация выполняется с использованием включающей нижней границы диапазона и исключающей верхней границы.

В результате транзакция отображается только в том случае, если ее сумма больше или равна минимальному значению диапазона и меньше максимального значения.

### Проверка

В ходе тестирования были проверены различные диапазоны суммы.

Получены следующие результаты:

| Сумма транзакции | Диапазон | Результат |
|------------------|----------|-----------|
| $100 | 99–100 | Не отображается |
| $100 | 100–100 | Не отображается |
| $100 | 100–101 | Отображается |
| $101 | 100–101 | Не отображается |
| $101 | 101–102 | Отображается |

### Анализ

Такое поведение соответствует модели диапазона **[min; max)** (нижняя граница включается, верхняя — исключается).

Но большинство пользователей ожидают, что обе границы диапазона будут включаться в поиск.

### Вопрос к требованиям

Является ли использование исключающей верхней границы диапазона ожидаемым поведением?

Если да, следует ли отразить эту особенность в пользовательском интерфейсе или изменить логику фильтрации таким образом, чтобы верхняя граница также включалась в диапазон поиска?

### Заключение

Наблюдение требует уточнения бизнес-требований.

## FINDING 7. Пользовательский интерфейс не позволяет выбрать диапазон суммы с одинаковыми минимальным и максимальным значениями

### Модуль

Лента транзакций

### Сценарий

Во время тестирования фильтра по диапазону суммы было установлено, что пользовательский интерфейс не позволяет вручную задать одинаковые значения минимальной и максимальной суммы (например, диапазон `100–100`).

Проверить данный сценарий удалось только с помощью автоматизированного тестирования.

### Проверка

Во время ручного тестирования установить диапазон `100–100` средствами пользовательского интерфейса не удалось.

Для проверки данного сценария использовался автоматизированный тест, который программно установил одинаковые значения минимальной и максимальной суммы.

### Анализ

Отсутствие возможности задать одинаковые границы диапазона ограничивает проверку поиска транзакции по точной сумме через пользовательский интерфейс.

При этом автоматизированное тестирование позволило проверить данный сценарий и выявить особенности работы фильтра.

### Вопрос к требованиям

Должен ли пользователь иметь возможность задавать диапазон с одинаковыми минимальным и максимальным значениями (например, `100–100`) для поиска транзакций на точную сумму?

### Заключение

Наблюдение требует уточнения требований к пользовательскому интерфейсу.

## FINDING 8. Сброс выбранных фильтров после обновления страницы

### Модуль

Лента транзакций

### Сценарий

Во время тестирования фильтрации транзакций было установлено, что после обновления страницы выбранные пользователем фильтры по диапазону дат и диапазону суммы автоматически сбрасываются к значениям по умолчанию.

### Проверка

Были выполнены следующие действия:

- выбран диапазон дат;
- установлен диапазон суммы;
- выполнено обновление страницы;
- проверено состояние фильтров после перезагрузки.

### Анализ

После обновления страницы:

- фильтр по диапазону дат сбрасывается в значение **Date: ALL**;
- фильтр по диапазону суммы сбрасывается в значение **Amount: $0 - $1,000**.

Такое поведение может быть реализовано намеренно, но в некоторых приложениях пользовательские параметры фильтрации сохраняются после обновления страницы для повышения удобства работы.

### Вопрос к требованиям

Является ли сброс выбранных фильтров после обновления страницы ожидаемым поведением?

Или приложение должно сохранять ранее выбранные параметры фильтрации после обновления страницы для повышения удобства пользователя?

### Заключение

Наблюдение требует уточнения бизнес-требований и ожидаемого поведения пользовательского интерфейса.
