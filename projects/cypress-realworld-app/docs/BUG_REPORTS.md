# BAG 1. API позволяет обновлять профиль пользователя невалидными значениями обязательных полей

## Описание

Эндпоинт обновления профиля пользователя принимает и сохраняет невалидные значения обязательных полей без выполнения серверной валидации.

При этом пользовательский интерфейс содержит проверки корректности ввода и не позволяет сохранить аналогичные данные через форму редактирования профиля.

Между клиентской и серверной валидацией наблюдается несогласованность.


## Предусловия

* Пользователь **Reyes.Osinski** существует.
* Пользователь успешно авторизован через API.


## Шаги воспроизведения

1. Выполнить `POST /login`.
2. Выполнить `PATCH /users/{id}`.
3. Передать, например, следующие данные:

```json
{
  "id": "M1ty1gR8B3",
  "firstName": "",
  "lastName": "",
  "email": "invalid-email",
  "phoneNumber": "",
  "defaultPrivacyLevel": "private"
}
```

4. Выполнить `GET /users/{id}`.
5. Авторизоваться через пользовательский интерфейс.
6. Открыть страницу **My Account**.


## Фактический результат

* API возвращает **204 No Content**.
* Невалидные данные успешно сохраняются.
* После входа через UI профиль содержит невалидные значения.
* При попытке повторно сохранить профиль через UI отображаются сообщения о некорректности данных.


## Ожидаемый результат

* API должен выполнять серверную валидацию входных данных.
* При передаче некорректных значений сервер должен возвращать ошибку валидации (например, **400 Bad Request** или **422 Unprocessable Entity**).
* Некорректные данные не должны сохраняться.


## Дополнительная информация

Проблема подтверждена для следующих сценариев:

* пустой `firstName`;
* пустой `lastName`;
* пустой `email`;
* некорректный `email`;
* пустой `phoneNumber`;
* некорректный `phoneNumber`.



## Примечание

UI и API используют разные правила проверки данных.

Через пользовательский интерфейс сохранить такие значения невозможно, тогда как API принимает и сохраняет их без ошибок. Это может привести к появлению неконсистентных данных в системе.



### Связанные ТС

- TC API ACCOUNT-2 
- TC API ACCOUNT-3
- TC API ACCOUNT-4
- TC API ACCOUNT-5 
- TC API ACCOUNT-6
- TC API ACCOUNT-7

# BAG-2. Авторизованный пользователь может изменять профиль другого пользователя через `PATCH /users/{id}`

## Описание

Авторизованный пользователь может изменить профиль другого пользователя, отправив PATCH-запрос с идентификатором чужого профиля.

Сервер не проверяет принадлежность изменяемого ресурса текущему пользователю и разрешает изменение данных другого пользователя.


## Предусловия

* Пользователь **A** существует (`M1ty1gR8B3`).
* Пользователь **B** существует (`uBmeaz5pX`).
* Пользователь **A** успешно авторизован.


## Шаги воспроизведения

1. Выполнить `POST /login` под пользователем **A**.

2. Отправить запрос:

   `PATCH /users/uBmeaz5pX`

3. Передать новые данные профиля пользователя **B**.

4. Выполнить `GET /users/uBmeaz5pX` или авторизоваться под пользователем **B**.

5. Открыть страницу **My Account**.


## Фактический результат

* API возвращает **204 No Content**.
* Изменения успешно применяются к профилю пользователя **B**.
* GET-запрос возвращает обновленные данные пользователя **B**.
* После авторизации под пользователем **B** через UI изменения отображаются на странице **My Account**.


## Ожидаемый результат

* Пользователь **A** не должен иметь возможность изменять профиль пользователя **B**.
* API должен вернуть ошибку авторизации (**401 Unauthorized**) или отказ в доступе (**403 Forbidden**).
* Данные пользователя **B** не должны изменяться.


### Severity: High


### Приоритет: High


## Примечание

Проблема говорит об отсутствии или некорректной реализации проверки прав доступа на серверной стороне.

Пользователь может изменять ресурсы, владельцем которых он не является, что нарушает принцип разграничения доступа и может привести к компрометации пользовательских данных.

### Связанные ТС

- TC API ACCOUNT-8

# BAG-3. API позволяет создавать банковский счет с невалидными значениями обязательных полей

## Описание

При исследовании GraphQL API создания банковского счета было обнаружено отсутствие серверной валидации входных данных.

API принимает и сохраняет некорректные значения обязательных полей, несмотря на то, что пользовательский интерфейс содержит проверки и не позволяет создать банковский счет с аналогичными данными.

Таким образом, между клиентской и серверной валидацией наблюдается несогласованность.

## Предусловия

* Пользователь авторизован в системе.


## Шаги воспроизведения

1. Выполнить GraphQL-мутацию `CreateBankAccount`.
2. Передать некорректные значения, например:

* пустой `bankName`;
* пустой `routingNumber`;
* пустой `accountNumber`;
* `routingNumber` короче допустимой длины;
* `accountNumber` длиннее допустимой длины;
* буквенные или специальные символы в полях `routingNumber` и `accountNumber`.

3. Выполнить запрос.
4. Авторизоваться через пользовательский интерфейс.
5. Открыть раздел **Bank Accounts**.


## Фактический результат

* GraphQL API успешно выполняет операцию создания банковского счета.
* Сервер возвращает **HTTP 200**.
* Банковский счет сохраняется с невалидными значениями.
* После входа через пользовательский интерфейс созданный банковский счет отображается в списке.


## Ожидаемый результат

* API должно выполнять серверную валидацию входных данных.
* При передаче некорректных значений сервер должен вернуть сообщение об ошибке валидации.
* Банковский счет с невалидными данными не должен создаваться.


## Дополнительная информация

Проблема подтверждена для следующих сценариев:

* пустой `bankName`;
* пустой `routingNumber`;
* пустой `accountNumber`;
* `routingNumber` меньше допустимой длины;
* `accountNumber` больше допустимой длины;
* буквенные символы в `routingNumber`;
* буквенные символы в `accountNumber`;
* специальные символы в `routingNumber`;
* специальные символы в `accountNumber`.

Во время проверки было установлено, что пользовательский интерфейс отображает ограничения для полей **Routing Number** и **Account Number** и не позволяет сохранить такие значения через форму создания банковского счета.

Но GraphQL API не применяет аналогичную серверную валидацию, что позволяет обойти ограничения пользовательского интерфейса и сохранить некорректные данные в системе.


### Severity: Medium


### Priority: Medium


### Связанные ТС

# BAG-4. API принимает отрицательную сумму транзакции, что приводит к некорректному изменению балансов пользователей и неконсистентному отображению операции в пользовательском интерфейсе

## Описание

API позволяет создать денежную операцию с отрицательным значением поля `amount`.

Сервер успешно принимает и сохраняет такую операцию, несмотря на отсутствие проверки корректности суммы.

При дальнейшем выполнении операции отрицательное значение приводит к некорректному изменению балансов пользователей. UI отображает такую операцию с некорректным знаком, и создает несогласованность между бизнес-логикой приложения и отображаемыми данными.

Проблема выявлена при работе с запросами на перевод (`request`), созданными с отрицательной суммой.


## Предусловия

* Пользователь успешно авторизован.
* В системе существует второй пользователь для выполнения денежной операции.


## Шаги воспроизведения

1. Авторизоваться через API.
2. Отправить запрос на создание денежной операции.
3. Передать отрицательное значение:

```json
{
  "amount": -550
}
```

4. Убедиться, что сервер успешно создает операцию.
5. При необходимости подтвердить созданный запрос на перевод через пользовательский интерфейс.
6. Проверить изменение балансов пользователей и отображение операции.


## Фактический результат

* API принимает отрицательное значение `amount`.
* Сервер успешно создает денежную операцию.
* Отрицательное значение сохраняется в системе.
* После выполнения операции балансы пользователей изменяются некорректно:
  * у пользователя, с которого должны списываться денежные средства, баланс увеличивается;
  * у пользователя, который должен получить денежные средства, баланс уменьшается.
* В истории операций пользовательского интерфейса сумма отображается с некорректным знаком.
* Пользовательский интерфейс также позволяет работать с операциями, содержащими отрицательную сумму.


## Ожидаемый результат

Возможны два корректных варианта поведения системы.

### Вариант 1 (предпочтительный)

API должно выполнять серверную валидацию значения `amount` и отклонять отрицательные суммы, возвращая ошибку валидации (например, **422 Unprocessable Entity**).

**или**

### Вариант 2

Если отрицательные значения допускаются бизнес-логикой приложения, все компоненты системы должны одинаково интерпретировать такие операции:

* корректно изменять балансы пользователей;
* корректно отображать сумму операции;
* не допускать выполнения операции в противоположном направлении.


## Дополнительная информация

Во время тестирования подтверждено следующее:

* API принимает значение `amount = -550`;
* сервер успешно создает операцию;
* отрицательное значение сохраняется;
* сервер не выполняет валидацию отрицательной суммы;
* при обработке операции происходит некорректное изменение балансов пользователей;
* история операций отображает сумму с некорректным знаком;
* аналогичное поведение подтверждено при работе с запросами на перевод (`request`) с отрицательной суммой.

Проблема свидетельствует об отсутствии серверной валидации отрицательных значений `amount` и некорректной обработке денежных операций в различных бизнес-сценариях приложения.


### Severity: High


### Priority: High


### Связанные ТС

- TC API REQUEST-6
- TC UI REQUEST-3
- TC API REQUEST E2E-1
- TC UI REQUEST E2E-1

# BAG-5. API не сохраняет дробную часть суммы при создании транзакции

## Описание

При создании новой транзакции API не поддерживает передачу суммы с дробной частью.

Если в поле `amount` передать значение с центами, сервер принимает запрос, но дробная часть не сохраняется - сумма автоматически приводится к целому числу.

При этом существующие транзакции в системе содержат суммы с дробной частью, что свидетельствует о несогласованности поведения приложения.


## Предусловия

* Пользователь успешно авторизован.
* В системе существует получатель перевода.


## Шаги воспроизведения

1. Авторизоваться через API.
2. Отправить запрос на создание новой транзакции.
3. Передать сумму с дробной частью, например:

```json
{
  "amount": 55.99
}
```

4. Выполнить запрос.
5. Проверить данные созданной транзакции через API или открыть историю транзакций в пользовательском интерфейсе.


## Фактический результат

* API успешно создает транзакцию.
* Сервер принимает дробное значение суммы.
* Дробная часть суммы не сохраняется - значение приводится к целому числу.
* В истории транзакций новая операция отображается без дробной части.


## Ожидаемый результат

Система должна работать единообразно.

Возможны два корректных варианта поведения:

1. API должно поддерживать создание транзакций с дробной частью суммы, если такие значения используются в системе.

**или**

2. Если дробные суммы не поддерживаются, API должно отклонять такие запросы и возвращать ошибку валидации, а не изменять переданное пользователем значение.


## Дополнительная информация

Во время тестирования установлено:

* существующие транзакции содержат суммы с дробной частью (например, `$36.56`);
* при создании новой операции через API дробная часть суммы не сохраняется;
* проблема подтверждена для различных значений (`55.01`, `55.33`, `55.99`);
* при создании запроса через пользовательский интерфейс значение `85.63` также сохраняется как `$85.00`;
* независимо от способа создания операции (API или UI) дробная часть суммы теряется.

Проблема свидетельствует о некорректной обработке дробных денежных значений серверной логикой и подтверждается как API-, так и UI-тестированием.


### Severity: Medium


### Priority: Medium

### Связанные ТС

- TC API REQUEST-7
- TC UI REQUEST-4

# BAG-6. API позволяет создать транзакцию для несуществующего получателя, после чего получение списка транзакций завершается ошибкой сервера

## Описание

При создании транзакции API не проверяет существование получателя.

Если в запросе передать несуществующий `receiverId`, сервер успешно создает транзакцию и возвращает **HTTP 200 OK**.

После создания такой записи последующий запрос на получение списка транзакций завершается ошибкой **500 Internal Server Error**, что свидетельствует о нарушении целостности данных.


## Предусловия

* Пользователь успешно авторизован.
* Используется несуществующий `receiverId`.


## Шаги воспроизведения

1. Авторизоваться через API.

2. Отправить запрос:

   `POST /transactions`

3. Передать несуществующий `receiverId`.

4. Убедиться, что сервер возвращает **HTTP 200 OK**.

5. Выполнить запрос:    `GET /transactions`


## Фактический результат

* API успешно создает транзакцию.
* Сервер возвращает **HTTP 200 OK**.
* При последующем запросе `GET /transactions` сервер возвращает **HTTP 500 Internal Server Error**.
* UI не может показать список транзакций.


## Ожидаемый результат

* API должно проверить существование получателя перед созданием транзакции.
* При передаче несуществующего `receiverId` сервер должен вернуть ошибку валидации (например, **400 Bad Request**, **404 Not Found** или **422 Unprocessable Entity**).
* Транзакция не должна создаваться.
* Запрос `GET /transactions` должен продолжать корректно работать и не приводить к внутренней ошибке сервера.


## Дополнительная информация

Во время тестирования установлено:

* `POST /transactions` возвращает **HTTP 200 OK** при несуществующем `receiverId`;
* после создания такой записи `GET /transactions` возвращает **HTTP 500 Internal Server Error**.

Проблема указывает на отсутствие проверки ссылочной целостности при создании транзакции и приводит к последующему сбою при обработке списка транзакций.


### Severity: High


### Priority: High


### Связанные ТС

# BAG 9. Запрос на перевод с нулевой суммой отображается в истории транзакций в формате, отличающемся от остальных денежных операций

## Описание

Пользовательский интерфейс позволяет создать запрос на перевод с суммой `0`.

После успешного создания операции сумма отображается в истории транзакций в формате `+0`, тогда как остальные денежные операции отображаются в денежном формате (например, `+$5.00`, `+$36.56`).

Таким образом, для нулевой суммы используется формат отображения, отличающийся от остальных денежных операций, что приводит к неконсистентному отображению данных в пользовательском интерфейсе.

## Предусловия

* Пользователь авторизован.
* В системе существует получатель перевода.

## Шаги воспроизведения

1. Авторизоваться в приложении.
2. Перейти к созданию нового запроса на перевод.
3. Выбрать получателя.
4. Ввести сумму `0`.
5. Заполнить поле **Description**.
6. Отправить запрос.
7. Открыть историю транзакций.

## Фактический результат

* Запрос успешно создается.
* В истории транзакций сумма отображается как `+0`.

## Ожидаемый результат

Сумма должна отображаться в едином денежном формате, используемом для остальных операций, например `+$0.00`.

Во время тестирования установлено:

* запрос успешно сохраняется;

# BAG-13. Сервер возвращает 500 Internal Server Error при передаче несовпадающих идентификаторов пользователя в URL и теле PATCH-запроса

## Описание

При обновлении профиля пользователя API не проверяет согласованность идентификаторов пользователя, переданных в URL и теле PATCH-запроса.

Если идентификатор в URL принадлежит одному пользователю, а в теле запроса указан идентификатор другого существующего пользователя, сервер успешно выполняет запрос и возвращает **HTTP 204 No Content**.

После этого последующие запросы к API завершаются ошибкой **500 Internal Server Error**, а пользовательская сессия становится неконсистентной.

---

## Предусловия

* Пользователь **Reyes.Osinski** успешно авторизован.
* В системе существуют два пользователя:
  * пользователь **A** (`M1ty1gR8B3`);
  * пользователь **B** (`uBmeaz5pX`).

---

## Шаги воспроизведения

1. Выполнить `POST /login`.
2. Отправить запрос:

   `PATCH /users/M1ty1gR8B3`

3. Передать тело запроса:

```json
{
  "id": "uBmeaz5pX",
  "firstName": "Pavel",
  "lastName": "QA",
  "email": "pavel@test.com",
  "phoneNumber": "123-456-7890",
  "defaultPrivacyLevel": "private"
}
```

4. Убедиться, что сервер возвращает **HTTP 204 No Content**.
5. Выполнить запрос:

   `GET /users/M1ty1gR8B3`

---

## Фактический результат

* `PATCH`-запрос возвращает **HTTP 204 No Content**.
* Последующий `GET`-запрос завершается ошибкой **500 Internal Server Error**.
* Сервер возвращает сообщение:

```
Failed to deserialize user out of session
```

---

## Ожидаемый результат

API должно корректно обработать ситуацию, когда идентификаторы пользователя в URL и теле запроса не совпадают.

Возможны несколько корректных вариантов поведения:

* использовать идентификатор пользователя из URL и игнорировать значение поля `id` в теле запроса;

**или**

* отклонить запрос и вернуть ошибку валидации, например:
  * **400 Bad Request**;
  * **409 Conflict**;
  * **422 Unprocessable Entity**.

Сервер не должен возвращать **500 Internal Server Error** и нарушать работу пользовательской сессии.

---

## Дополнительная информация

Во время тестирования установлено:

* API не проверяет согласованность идентификаторов пользователя;
* `PATCH`-запрос успешно выполняется при несовпадающих `id`;
* после выполнения такого запроса последующие обращения к API завершаются ошибкой **500 Internal Server Error**;
* сервер возвращает сообщение `Failed to deserialize user out of session`.

Проблема свидетельствует об отсутствии проверки согласованности входных данных и приводит к нарушению корректной работы серверной сессии.
